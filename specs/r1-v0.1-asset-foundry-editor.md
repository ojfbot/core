## Problem statement

asset-foundry today is a build-time pipeline: a CLI invocation (`pnpm gen-asset`) runs the LangGraph orchestrator (4 sub-agents: WorldDesigner, AssetSculptor, MaterialArtist, SceneAssembler), Blender (subprocess) executes a bpy script, the deterministic Validator gates the output, and validated `.glb` artifacts auto-sync to consumer games' `public/assets/`. There is no UI surface and no interactive feedback loop.

The user wants a Frame MF web UI that turns this into an **interactive asset editor**:
- Switch between active games at the top level.
- Per-game: browse the asset library (known `.glb`s + new entries from the manifest).
- Click an asset: see a 3D Three.js preview.
- Use a multimodal chat (text + drag-drop reference images) to give iteration feedback.
- The orchestrator runs through a *production* Blender MCP path, regenerates the `.glb`, the preview reloads, and the game's `public/assets/` updates within seconds.

v0.1 is **local-dev only**. The shared codebase will later support a public deployment (read-only inspector + LLM-feedback-to-GitHub-issues), but that's R1 v0.2. URL-as-reference and `.blend` file uploads are deferred (also v0.2 or v0.1.x).

### Bounded-context impact (CONTEXT.md)

- **Agent Graph**: orchestrator gets a `chatContext` field on its `State`; AssetSculptor's prompt accepts text feedback + image content blocks (via frame-agent).
- **Workflow Engine**: `/frame-dev` registers asset-foundry's web + API + Blender MCP per Sprint M.
- **Shell + Host Composition**: asset-foundry-ui added to app registry; `RigProfile` transitions `non-frame` → `frame`.
- **UI Components**: new patterns — drag-drop image upload zone, 3D preview pane (Carbon Tile-wrapped Three.js scene), game-switcher chrome.
- **Gas Town Governance**: `fnd-` bead prefix already reserved (ADR-0052); RigProfile transition is a CONTEXT.md update.

New aggregates (CONTEXT.md update needed in this sprint):
- **AssetLibrary** — per-game collection of assets with version history and validation status. Source: `dist/<game>/*.validation.json`.
- **AssetChatSession** — the unit of a one-shot regenerate-with-context turn (NOT a long-running thread; v0.1 chat history is UI-only, not state-persisted).

### Sprint Z — frame-agent vision content blocks (PREREQUISITE)

**Repo:** `shell` (NOT asset-foundry).
**Why first:** Sprint D's "drag a reference image into chat" acceptance criterion needs frame-agent to forward image content blocks to Anthropic. Today it's text-only.

**Tasks:**
- Extend `ChatRequestSchema` at `shell/packages/frame-agent/src/routes/chat.ts:8-24`. Current shape:
  ```ts
  message: string,
  context: { activeAppType, instanceId, threadId, instances }?,
  conversationHistory: { role, content: string }[]?
  ```
  Extend to allow `content` to be `string | ContentBlock[]` where `ContentBlock` is the Anthropic SDK shape: `{type: 'text', text: string} | {type: 'image', source: {type: 'base64', media_type: 'image/jpeg' | 'image/png' | 'image/webp' | 'image/gif', data: string}}`.
- Wire the multipart content through to the Anthropic message construction (currently a single-string user message; becomes an array).
- Wire the existing `/api/inspect` `screenshot` field per the `// TODO: pass to model once vision support is wired end-to-end` comment.
- ADR documenting this in shell's `decisions/`. Lives in the shell repo.
- No changes to existing sub-app callers; old `content: string` shape stays valid.

**Acceptance:** `curl -X POST localhost:4001/api/chat -d '{"message":"what's in this image", "conversationHistory":[{"role":"user","content":[{"type":"text","text":"see attached"},{"type":"image","source":{"type":"base64","media_type":"image/jpeg","data":"<b64>"}}]}]}'` returns a vision-aware response.

### Sprint B — multi-game manifest schema + asset library tab

**Tasks:**
- **Schema migration (asset-foundry/manifest):**
  - `manifest/world.yaml` → `manifests/<game>/world.yaml`. Initial migration: move existing world.yaml to `manifests/beavergame/world.yaml`.
  - Each manifest declares a top-level `consumer_path: "../<game>/public/assets/"` field.
  - Update `manifest/schema.ts` Zod schema to add `consumer_path` (default falls back to derived value from manifest dir).
  - Update `manifest/load.ts:findProp(manifest, propId)` to take a manifest argument (already does, but the manifest loader becomes per-game).
- **CLI per-game args:**
  - `scripts/gen-asset.ts <game> <prop_id>` — first arg is the game.
  - `scripts/validate.ts <game>` — validates one game's assets.
  - `scripts/validate-manifest.ts <game>` — validates the per-game manifest.
- **Per-game dist namespace:**
  - `dist/<game>/<id>_v<N>.glb` and `dist/<game>/<id>_v<N>.validation.json`.
  - **Migration ordering (one-time script):** (1) snapshot existing `dist/*.glb` + `*.validation.json` to `/tmp/asset-foundry-pre-migration/`. (2) `mkdir -p dist/beavergame/`. (3) `git mv dist/*.glb dist/beavergame/` and same for `.validation.json`. (4) Run `pnpm validate beavergame` — must pass with the new paths. (5) Update consumer game's `public/assets/` references if they were absolute. (6) Commit. Failure mode: if validate fails after move, restore from snapshot.
- **Library tab UI:**
  - `packages/browser-app/src/components/Library.tsx` — Carbon `Tile` per asset. Each tile shows `id`, `tri_count`, validation status (✓/✗), last-built timestamp.
  - Distinguishes "known" (has `.glb` + passing validation) from "new/draft" (in manifest but no `.glb` yet).
  - Click a tile → selects asset (Sprint C uses this).

**Acceptance:** `pnpm validate-manifest beavergame` passes. `pnpm gen-asset beavergame birch_sapling` regenerates `dist/beavergame/birch_sapling_v2.glb` (Sprint B keeps subprocess Blender; MCP comes in Sprint D). UI library tab shows all 5 known assets with metadata.

### Sprint D — multimodal chat + orchestrator wiring + Blender MCP

**Tasks:**
- **Chat panel UI (`packages/browser-app`):**
  - `AssetChatPanel.tsx` in 320px side panel (per scaffold-frame-app pattern).
  - Text input + drag-drop image zone (accepts .jpg/.png/.webp; rejects others with toast).
  - On submit: serialize images to base64, POST to `/api/chat` with `{message, referenceImages[], game, prop_id}`.
  - SSE-stream the orchestrator progress back; render as turn updates.
- **API chat route (`packages/api`):**
  - `POST /api/chat` validates payload (Zod).
  - **Image transport: in-memory only.** Browser sends multipart form (or base64 JSON) → API decodes to a `Buffer` per image, validates MIME by sniffing magic bytes, forwards as base64 content blocks to frame-agent. **Never persisted to disk.** (Choice: avoids `/tmp` cleanup complexity + matches frame-agent's content-block API.)
  - Invokes the orchestrator with `chatContext = {prompt: message, referenceImages: [{mediaType, base64}]}`.
  - Streams agent progress via SSE (per existing langgraph-patterns.md SSE pattern).
- **Orchestrator extension (`packages/agent-graph`):**
  - State adds `chatContext?: ChatContext` field. **Per CONTEXT.md §2 invariant ("every field has a reducer"), define a reducer in `state/schema.ts` — recommend last-write-wins for `chatContext` since it's per-turn input, not accumulated state.**
  - AssetSculptor sub-agent's prompt accepts `chatContext`. Reference images become content blocks in the LLM call (forwarded through frame-agent — depends on Sprint Z).
  - The route layer in frame-agent at `shell/packages/frame-agent/src/routes/chat.ts:35` is currently `metaOrchestrator.route(message, context, conversationHistory)` — Sprint Z must extend the orchestrator's signature to accept structured content arrays, not just route-layer schema changes.
  - Existing CLI path (`gen-asset.ts`) sets `chatContext: undefined`; behaves as today.
- **Extend the Blender MCP bridge:**
  - `asset-foundry/src/blender/mcp-bridge.ts` already exists as the subprocess Blender bridge with the `FOUNDRY_USE_MCP` toggle in its comment header (no MCP client wired today). Sprint D **extends** this file: add an MCP client path alongside the existing `runBlenderScript()` subprocess code.
  - MCP client connects to local Blender MCP server (started by `frame-dev.sh` per Sprint M, on port TBD; reference `asset-foundry/scripts/install-blender-mcp.sh` which already detects Blender via `BLENDER_BIN` / `.blender-version`).
  - Bridge interface stays consistent: `executeBpyScript(script: string, outputPath: string): Promise<{success: boolean, summaryJson?: object, error?: string}>`. Internally branches on `FOUNDRY_USE_MCP` env: `1` → MCP path, otherwise subprocess (the existing today-default).
  - In v0.1 dev mode, `FOUNDRY_USE_MCP=1` is set by `frame-dev.sh` so the editor uses MCP; CLI users keep the subprocess default for backwards compat.
- **Auto-sync pipe-through:**
  - On successful orchestrator run + validator pass, the existing post-step that copies `dist/<game>/<id>_v<N>.glb` → `<consumer_path>/<id>_v<N>.glb` runs.
  - The browser AssetPreview polls the asset endpoint with cache-busting; reloads on detected change.

## Security section

The prompt-feature security guide flags this work as **moderate to high blast radius** because it introduces:

1. **Blender MCP execution path.** Anything the orchestrator generates as a bpy script runs with the user's Blender process privileges (filesystem write, subprocess spawn potential). Mitigations:
   - MCP server binds localhost only; verify in mcp-bridge.ts.
   - bpy script generation is bounded by the AssetSculptor sub-agent's prompt + fixture patterns (`_lib.py` helpers). No user-supplied raw Python.
   - Output paths constrained to `dist/<game>/`.
   - **Out of scope (deferred to v0.2):** `.blend` file uploads — these would let user-supplied Blender state run; explicit security review required at that time.

2. **User-supplied images via chat.**
   - MIME validation on upload (sniff actual bytes; don't trust extension).
   - Size cap (configurable; default 10MB per image, 4 images per turn).
   - Stored in `/tmp/asset-foundry-chat/<request-id>/` with 5min TTL; never persisted.
   - Forwarded to Anthropic via frame-agent vision blocks (Sprint Z); not stored on Anthropic side per their API guarantees.
   - **Prompt-injection risk:** an image with embedded text instructing the model to do something harmful (jailbreak via vision). Mitigation: AssetSculptor's prompt explicitly frames images as *visual references*, not instructions; output remains constrained to bpy-script generation; validator catches anything off-spec.

3. **API surface.**
   - `/api/asset/<game>/<id>` — path-traversal hardened (regex `^[a-z][a-z0-9_-]*$` for both params).
   - `/api/chat` — Zod-validated payload; size cap on `message` (4KB) and `referenceImages` (40MB total).
   - `/api/library/<game>` — read-only, lists from `dist/<game>/*.validation.json` only.
   - **No auth in v0.1** (local-dev only). `pnpm dev` binds 3036 to `localhost`. v0.2 adds auth.

4. **Consumer game's `public/assets/` write surface.**
   - Existing `gen-asset.ts` already writes here; not new.
   - Output path is constrained by the per-game `consumer_path` field in the manifest (which is human-edited, not user-supplied at runtime).

## Open questions

1. **Image storage: RESOLVED.** In-memory `Buffer` decoding in API → base64 content blocks to frame-agent → never persisted to disk. (See Sprint D "API chat route" for the implementation note.)
2. **Orchestrator one-shot vs multi-turn.** v0.1 spec: one-shot per chat message. Conversation history is UI-only. Confirm with user.
3. **Blender MCP server lifecycle.** Started by `frame-dev.sh` (Sprint M) as a long-running localhost service, or per-orchestrator-run subprocess? Recommend: long-running service via Sprint M.
4. **Port allocation.** browser-app 3035, API 3036. Confirm against `frame-dev.sh` allocations and shell's app registry.
5. **`gen-asset.ts` CLI.** Keep for CI/scripted regen, or deprecate in favor of API-only? Recommend keep; CI smoke tests need it.
6. **Validation failure surfacing in chat.** Surface inline (recommend) vs silent retry vs abort and require manual rerun?
7. **Asset versioning / history UI.** v0.1 just shows latest. Should the Library tile show version count? Recommend: yes, badge.
8. **MCP protocol details.** Anthropic MCP standard? Custom JSON-RPC over TCP? Pick one in mcp-bridge.ts implementation.
