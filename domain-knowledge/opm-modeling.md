# OJF-OPL — Object-Process Modeling profile for the ojfbot fleet

Reference for the `/opm` skill and any agent reading or writing `opm/system.opl` files.
Profile of ISO/PAS 19450 OPL (Dov Dori's Object-Process Methodology). Design intent: **one fact
per line, in controlled English, checkable by grep and by LLM alike.** See ADR
`opm-inspectability-layer` and selfco `wiki/sources/opm-deep-research.md` for rationale.

## The ontology in 60 seconds

Everything is one of two **things**:

- **Object** — something that *exists* (a bead, an article, a telemetry ledger, a repo, a person).
  Objects may have **states** (`draft`/`published`, `open`/`closed`). Rendered as rectangles.
- **Process** — something that *happens*, always named as a gerund ("Article Generating"). A
  process is only a process if it transforms at least one object: creates it, consumes it, or
  changes its state. Rendered as ellipses.

Two orthogonal attributes, with fleet defaults so most sentences omit them:

- **Essence**: `informatical` (default — code, data, docs) vs `physical` (people, machines).
- **Affiliation**: `systemic` (default — part of the repo/fleet) vs `environmental` (GitHub,
  Anthropic API, Notion — context we don't control).

**Links** carry all remaining meaning. Enablers are *present but untransformed*; the
agent/instrument split is the human-in-the-loop boundary and is the profile's most load-bearing
distinction: **agent = intelligent enabler that decides; instrument = tool that is used.**

## Sentence templates (OJF-OPL v0.1)

Naming: Things are Capitalized Noun Phrases; processes end in *-ing*. One sentence per line,
ending with a period. `#` starts a comment. Bold-reserved words below are literal.

### Things and states
| Fact | Template |
|---|---|
| Declare non-default essence/affiliation | `Jim is physical.` · `GitHub API is environmental.` |
| States | `Article can be draft or published.` |
| Initial state | `Draft is initial.` (optional) |

### Structural links
| Relation | Template |
|---|---|
| Aggregation | `Pipeline consists of Collecting, Generating, and Publishing.` |
| Generalization | `Skill Telemetry Record is a Ledger Line.` |
| Exhibition (attribute) | `Article exhibits Status.` |
| Instantiation | `ADR-0098 is an instance of ADR.` |
| Tagged (free) | `Cockpit reads from Dolt.` (any verb phrase; use sparingly) |

### Procedural links
| Relation | Template | Semantics |
|---|---|---|
| Agent | `Jim handles Approving.` | intelligent enabler — human/LLM judgment in the loop |
| Instrument | `Generating requires Claude API.` | non-intelligent enabler — tool/API/file used |
| Consumption | `Generating consumes Commit Context.` | input destroyed/claimed |
| Result | `Generating yields Article.` | output created |
| Effect | `Linting affects Wiki.` | changes object, unspecified how |
| State change | `Publishing changes Article from draft to published.` | the precise form of effect |
| Condition | `Publishing occurs if Approval is granted.` | guard |
| Event trigger | `Cron Firing triggers Generating.` | object/state-change starts process |
| Invocation | `Generating invokes Verifying.` | process starts process |

### Decomposition (the OPD tree)
| Mechanism | Template |
|---|---|
| In-zoom (process) | `Generating zooms into Collecting, Drafting, and Synthesizing.` — subprocesses run in listed order |
| Unfold (object) | covered by `consists of` |

Model files are organized as one `## SD` section (top-level System Diagram: the 5–9 things that
matter) plus one `## SD1.n — <Process> in-zoom` section per zoomed process. Every thing mentioned
in a child section must appear in, or descend from, a parent section.

### Provenance anchors (OJF extension — not ISO)
Any sentence may end with a bracketed anchor before the period's line end:
`Generating yields Article. [src: src/generate-article.ts]`
`/opm lint` verifies anchored paths exist in the working tree. Skills are anchored as
`[skill: daily-logger]`, ADRs as `[adr: <slug>]`.

## File convention

```
opm/
  system.opl    # the model — source of truth, hand-reviewed, LLM-authored
  system.md     # deterministic Mermaid rendering, regenerated by /opm render — do not hand-edit
```

## Rendering rules (OPL → Mermaid)

One `flowchart TD` block per `##` section. Deterministic — same .opl in, same .md out:

| OPL | Mermaid |
|---|---|
| Object X | `X[X]` — add `<br/>{s1, s2}` to the label if states declared; environmental objects get dashed style `style X stroke-dasharray: 5 5` |
| Process P | `P(P)` (rounded) |
| `P consumes X.` | `X --> P` |
| `P yields Y.` | `P --> Y` |
| `P affects Z.` / state change | `P -- affects --> Z` / `P -- s1→s2 --> Z` |
| `A handles P.` | `A -. agent .-> P` |
| `P requires I.` | `I -. instrument .-> P` |
| `P occurs if C.` / triggers / invokes | `C -. if .-> P` / `C -. triggers .-> P` / `P -. invokes .-> Q` |
| `A consists of B, C.` | `B -- part of --> A` etc. |
| `B is an A.` | `B -- is a --> A` |
| in-zoom | child section's own flowchart; parent node linked by heading anchor |

## ojfbot ↔ OPM mapping (the Rosetta row)

| ojfbot term | OPM reading |
|---|---|
| bead, article, ADR, ledger/JSONL, vault page, PR | object (stateful — their `status:` fields are OPM states) |
| skill run, pipeline phase, cron routine, hook, daemon | process |
| Jim; a Claude session/subagent | agent (intelligent enabler) |
| CLI, API, MCP server, template file, model checkpoint | instrument |
| GitHub, Anthropic API, Notion, Vercel | environmental object |
| gated-slice gate; OPAV disposition | condition link on the gated process |
| orchestrate layers / slice decomposition | in-zooming |

## Authoring discipline

- **Model the true system, not the aspiration.** An OPL line is a claim; if it isn't checkable or
  currently true, it doesn't go in (aspirations belong in ADRs/roadmaps). Same rule as
  morning-cockpit's "Honest gaps".
- **Small SD.** 5–9 things at top level; push detail into in-zooms. Clutter is the known failure
  mode of one-diagram-kind notations.
- **Prefer the precise link.** `changes X from a to b` over `affects X`; `affects` is a smell that
  states are missing.
- **Every process transforms something.** A "process" with only enabling links is a misdrawn
  object or a missing `yields`.
- **Shadow-first.** `/opm lint` reports; it does not gate. Promotion to a CI gate is a RIDM
  decision after pilot data (ADR `opm-inspectability-layer` §Rollout).
