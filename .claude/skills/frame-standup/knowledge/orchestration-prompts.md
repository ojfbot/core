# Orchestration Prompt Templates

Reference for `/frame-standup` Step 7. Defines the Layer 1, 2, and 3 prompt
templates used by the progressive decomposition engine (`/orchestrate`).

---

## Layer 1 — Per-App Orchestrator Prompt

Generated by `/frame-standup` Step 7. One per app with priorities today.
The agent receiving this prompt plans and decomposes — it does NOT write code.

````markdown
### Layer 1 Orchestrator — <repo-name>

**Mode:** Plan-first orchestrator (decompose only — do NOT implement)
**Working directory:** ~/ojfbot/<repo-name>

**Today's priorities for this app:**
1. <P0/P1: description> · specificity: high|medium|low
2. <description> · specificity: high|medium|low
3. <description> · specificity: high|medium|low

**Context files to read:**
- `~/ojfbot/core/domain-knowledge/<repo>-architecture.md` — architecture overview
- `~/ojfbot/<repo>/.claude/standup.md` — current blockers and priorities
- `~/ojfbot/<repo>/CLAUDE.md` — development setup and conventions

**Activity context:** <activityType> day · <N> open actions · <N> recently closed
**Daily-logger post:** <URL>

**Your job:**
1. Read the context files above
2. For each priority, produce a decomposition:
   - What discrete tasks are needed?
   - What order must they execute in? (dependency graph)
   - What files does each task touch? (specific paths, not directories)
   - What is the verification criterion for each task? (test command or check)
   - What context does each task need? (file paths only — no architecture docs)
3. Output: A numbered list of Layer 2 task prompts, each self-contained

**For high-specificity goals:** Decompose directly into file-level tasks.
**For medium-specificity goals:** First identify the target state (read relevant
ADR or spec), then decompose.
**For low-specificity goals:** Produce a `/plan-feature` prompt instead of
decomposing — the goal needs user refinement first.

**Relevant skills (suggest to user if applicable):**
{{SKILL_SUGGESTIONS_L1}}

**Do not write code. Do not open PRs. Only plan and decompose.**
````

> **Generating L1 skill suggestions:** Before building a Layer 1 prompt, the
> orchestrator runs:
> ```bash
> node "$CLAUDE_PROJECT_DIR/scripts/hooks/suggest-skills.mjs" \
>   --tags "<priority-tags>" --phase "<phase>" --layer=1 --format=oneline
> ```
> and replaces `{{SKILL_SUGGESTIONS_L1}}` with the output. If the engine
> returns nothing, omit the "Relevant skills" section entirely.

## Layer 2 — Task Decomposition Prompt

Generated by Layer 1 output. One per discrete task. The agent refines the
task into implementation-ready file-level instructions.

````markdown
### Layer 2 Task — <task-title>

**Mode:** Task decomposition (refine into implementation steps — do NOT implement)
**Working directory:** ~/ojfbot/<repo-name>
**Depends on:** <task IDs that must complete first, or "none">

**Task description:**
<2-3 sentences from Layer 1 describing what this task accomplishes>

**Files to read:**
- `<specific source file>` — <why>
- `<specific test file>` — <why>
- `<ADR or spec file>` — <target state definition>

**Parallel implementation (if applicable):**
- `<path to similar implementation in another repo>` — follow this pattern

**Your job:**
1. Read the listed files
2. Describe the exact changes needed (function signatures, import additions,
   config modifications)
3. Define the verification command (e.g. `pnpm vitest run <test-file>`)
4. Output: A single Layer 3 execution prompt

**Relevant skills:**
{{SKILL_SUGGESTIONS_L2}}

**Do not write code. Describe what to change with enough detail that an
implementation agent can execute without reading architecture docs.**
````

> **Generating L2 skill suggestions:** Before building a Layer 2 prompt, the
> orchestrator runs:
> ```bash
> node "$CLAUDE_PROJECT_DIR/scripts/hooks/suggest-skills.mjs" \
>   --tags "<task-tags>" --layer=2 --after="/<previous-skill>" --format=oneline
> ```
> and replaces `{{SKILL_SUGGESTIONS_L2}}` with the output.

## Layer 3 — Execution Prompt

Generated by Layer 2 output. The agent implements the change in an isolated
worktree, runs tests, and opens a PR.

````markdown
### Layer 3 Execution — <change-title>

**Mode:** Implementation (write code, run tests, open PR)
**Working directory:** ~/ojfbot/<repo-name>
**Isolation:** worktree (use EnterWorktree for independent changes)

**What to change:**
1. In `<file-path>`: <exact description of change>
2. In `<file-path>`: <exact description of change>
3. ...

**Expected behavior after change:**
<What should be true when the change is correct>

**Verification:**
```bash
<test command>
```

**PR title:** `<type>(<scope>): <description>`
**PR base:** `main`

**On failure — suggest these skills:**
{{SKILL_SUGGESTIONS_L3_FAILURE}}

**Do not:**
- Change files not listed above
- Add features beyond what is specified
- Skip the verification step
````

> **Generating L3 failure-path suggestions:** Before building a Layer 3 prompt,
> the orchestrator runs:
> ```bash
> node "$CLAUDE_PROJECT_DIR/scripts/hooks/suggest-skills.mjs" \
>   --tags "debug,test" --layer=3 --format=oneline
> ```
> and replaces `{{SKILL_SUGGESTIONS_L3_FAILURE}}` with the output. This section
> only appears in the failure path — do not clutter the execution context with
> suggestions the agent won't need if everything passes.

## Context budget summary

| Layer | What it receives | What it does NOT receive |
|-------|-----------------|------------------------|
| L1 | Architecture doc, CLAUDE.md, standup.md, priorities, skill suggestions (phase/tag-matched) | Other apps, personal-knowledge, full roadmap |
| L2 | Specific source files, ADR/spec, parallel implementation, skill suggestions (chained via `--after`) | Architecture overview, standup, roadmap |
| L3 | Exact files to change, expected behavior, test command, failure-path skill suggestions | Architecture docs, other tasks, roadmap |

The key principle: **each layer receives only what it needs to do its job.**
Passing too much context dilutes focus and wastes tokens. Passing too little
causes the agent to guess or read unnecessary files.
