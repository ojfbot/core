# Synthesis Guide

The synthesis phase produces the final revised document by incorporating council feedback. This is the most critical phase — it must improve the document without compromising its integrity.

## Input

- Original draft (full text)
- N council critiques (one per persona, each with Questions / Gaps / What lands)

## Output shape

Same document structure as the original. Not a new document, a revised one.

## Rules

### 1. Answer questions where the information exists

If a persona asks "Is this live?" and the context tells you it is: add one clear sentence.
If a persona asks "What's the blast radius?" and the PR diff shows the scope: state it explicitly.

Only answer if the information is actually available in context. Do not fabricate.

### 2. Acknowledge gaps honestly

If reviewers asked something you cannot answer: a single honest sentence is better than silence.

Good: "The monitoring dashboard is not yet configured — this is a known gap for Phase 2."
Bad: Skipping the gap entirely.
Bad: Adding vague hedges ("we're working on it" without specifics).

The target reader is a technical evaluator. Honest gaps build trust. Hidden gaps destroy it.

### 3. Sharpen what landed

If multiple personas noted the same thing "worked" — make it more prominent. Move it higher if needed. Add one sentence of context if the framing needs it.

Do not cut things that landed. Ever.

### 4. Preserve structure

Original section headers stay. The final document must have the same sections in the same order as the original. If a section is weak, strengthen its content — do not remove it or merge it with another.

### 5. Length constraint

Final document should be 100–150% of original word count. If you're exceeding 150%, you are adding new content rather than revising existing content. Cut instead.

## What NOT to do

- Do not add marketing language to cover gaps ("exciting next steps", "powerful platform")
- Do not add bullet points where the original had prose
- Do not add a "Summary of changes" header to the output document itself
- Do not repeat council critiques in the document body
- Do not attribute changes to any persona in the document text

## Change log format

After the synthesized document, append:

```
## Change log
- Added: [what and why — reference the reviewer question or gap]
- Removed: [what was cut and why]
- Preserved: [what was kept explicitly due to council praise]
```

The change log is NOT part of the document — it follows the document and is for reviewer reference only.

## Tone calibration

The document's voice must match the original. If the original is technical and direct, the revision must be too. Council synthesis should not make the document sound like it was written by committee.

One voice, improved.
