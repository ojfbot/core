# Research — headless component contract precedent survey

**Date:** 2026-07-23 · **Ticket:** wayfinder #276 (map `operating-surface-bonded-pair`) · **Method:** deep-research harness, ONE cycle (5 angles → 23 sources → 112 extracted claims → top 25 adversarially verified, 3 votes each: 24 confirmed, 1 refuted, 0 unverified; 105 agents) · **Feeds:** #277 (frame-ui-components refactor path), #278 (native design system bootstrap) · **ADR context:** adr:headless-components-with-design-language-adapters

## Verdict (one paragraph)

The surveyed systems converge on a four-layer headless contract that transfers directly to the fleet: **(1) behavior as framework-agnostic statecharts** (Zag.js is the strongest precedent — one machine, thin per-framework adapters; exactly the shape any-stack Tier B needs); **(2) an anatomy/part vocabulary** bridged to the DOM via a `connect()`-style layer emitting prop getters, ARIA semantics, and machine-readable `data-part`/`data-state` hooks, anchored to WAI-ARIA APG; **(3) a skin boundary below all visual presentation** (Radix, Zag, Base UI ship zero design-language CSS; Base UI's render-prop + typed-state pattern is the precedent for per-design-language markup replacement); **(4) a token layer** — DTCG format (first stable release 2025.10) as the interchange, with Material 3's reference→system→component indirection as the layering model where skinning happens by remapping the system layer over a fleet-owned neutral role vocabulary.

## Verified findings (all 3-0 unless noted)

1. **Statechart core is the portable unit.** Zag models every component interaction as a statechart defined independently of any UI framework; the machine is the reusable behavior unit. (zagjs.com/overview/introduction · /whats-a-machine · github.com/chakra-ui/zag)
2. **Write-once/adapt-everywhere is proven.** One behavior definition exposed to React/Solid/Vue/Svelte via thin adapters (`useMachine` + `connect` + `normalizeProps`). Caveat: non-adapter stacks (three.js, Python-served pages, vanilla TS) must write the normalizeProps/render-loop glue — small but non-zero fleet work.
3. **The contract surface is a connect layer emitting anatomy-part prop getters** (`getRootProps()`, `getControlProps()`, …), each part tagged `data-part`/`data-scope`. The anatomy vocabulary is the piece the fleet defines once and holds stable — every skin and adapter addresses parts by these names.
4. **Accessibility lives in the shared contract, not the skin.** Zag's machines+connect own keyboard, focus, and ARIA, modeled on WAI-ARIA APG, with 60+ cross-framework Playwright e2e suites. Qualifiers: APG is a W3C guidance Note, not a Recommendation; a11y transfers only if consumers spread props onto the right parts; a skin's CSS can still degrade contrast/focus.
5. **All three mature systems draw the skin boundary below all visual presentation** — zero design-language CSS shipped (Radix/Zag/Base UI). Necessary but not sufficient for multi-language skinning: anatomy naming and token layering are separate surfaces skins inherit.
6. **`data-state`/`data-part` attributes are the cross-stack styling hook.** CSS-addressable (`[data-part="trigger"][data-expanded]`, `[data-state="open"]`) with no framework runtime — the strongest Tier B-relevant pattern: a skin authored purely as CSS against a frozen vocabulary is consumable by any HTML-emitting stack.
7. **Base UI is the precedent for per-design-language markup replacement** *(medium confidence; one 2-1 vote)*: render prop per part, two-rule composition contract (forward ref, spread props), typed state object across the boundary (`render={(props, state) => …}`). React-specific mechanics; the idea (typed state across the boundary alongside data-attributes) transfers.
8. **A pinnable token interchange exists.** DTCG format reached first stable version 2025.10 (2025-10-28); vendor-agnostic JSON designed for cross-tool/platform theming. Caveats: theming/modes live in the separate **Resolver Module, still a draft**; DTCG is community-group-stable, not standards-track. Pin the Format Module; watch-list Resolver.
9. **Material 3 supplies the token-layering model.** Exactly three token classes (reference → system → component), indirection pointing downward, theming located at the system layer. Precision: Material's documented contexts are intra-Material (light/dark, contrast, density, RTL) — a cross-design-language swap additionally requires the fleet to own a **neutral role vocabulary** at the system layer; `md.sys.*` roles are Material's own.
10. **Recommended fleet contract shape** *(synthesis, medium confidence — each element unanimously verified, the composition is the researcher's)*: (a) dependency-free statechart core, APG-anchored, cross-stack e2e per component; (b) connect-style adapter seam — React adapter first plus a vanilla-DOM adapter for FastAPI pages / three.js overlays / TS games; (c) frozen `data-part`/`data-state` vocabulary so each skin (Carbon/HIG/Material/native) is primarily a stylesheet + optional markup override; (d) DTCG-format tokens in a reference→system→component chain, skins swapping at the system layer over a fleet-owned role vocabulary; (e) an explicit **functional-vs-design-language style split** so positioning/visually-hidden styles live in the core without violating the "unstyled" promise.

## Refuted (0-3) — and load-bearing

- "Radix's contract excludes even functional styles." **False**: Radix Popper-positioned components apply hardcoded inline positioning (issues #2908/#1253); Zag's connect emits functional inline styles. Fleet contract language must not promise zero shipped styles — codify the functional/design-language split instead.

## Coverage gaps (honest limits of this ONE cycle)

React Aria / react-aria-components, Headless UI, Ark UI's own layer above Zag, Lit, Shoelace/Web Awesome, and Open UI's anatomy work produced no claims that reached 3-vote verification — the comparison is strongest for Radix/Zag/Base UI/DTCG/Material and **silent on the web-components and React-Aria families**. Open UI's anatomy effort bears directly on the data-part vocabulary recommendation. Nearly all evidence is first-party vendor docs (fine for architectural self-description; "WAI-ARIA compliant" is self-attestation). No verified precedent ships one behavior core under four genuinely distinct corporate design languages — Ark/Chakra/Park over Zag is multi-theme more than multi-design-language; the multi-skin recommendation is an extrapolation.

## Open questions (candidate inputs to #277/#278 grills)

1. Should the fleet's `data-part` vocabulary align with Open UI's / React Aria's / Ark's part naming rather than inventing its own?
2. Concrete cost of a vanilla-DOM adapter for Tier B targets — FastAPI-served pages with minimal JS, three.js scenes where "parts" may not be DOM nodes, canvas games?
3. Can one neutral system-token role vocabulary span Carbon, HIG, Material, and native without lowest-common-denominator loss (HIG vibrancy/materials and Carbon spacing have no Material equivalents)? Does the draft DTCG Resolver modifier-axis model express a design-language axis, not just light/dark?
4. Do web-component systems (Lit, Shoelace) with `::part()`/custom-property surfaces offer a better Tier B delivery vehicle than machine+adapter — should the skin contract target `::part()` alongside data-part CSS?
